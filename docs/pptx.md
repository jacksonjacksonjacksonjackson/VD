### 4.2 Template discovery (no hardcoded paths)
- Try in this order:
  1. `template_path` param (if provided)
  2. A config/env variable you already use (e.g., `OPTONY_PPTX_TEMPLATE`)
  3. Repo search for `*.potx` containing the required **named shapes** (Title, Subtitle, FooterDate, FooterClient, KPI_1..KPI_6, Chart_1, Table_1). If none found, generate a **temporary** minimal template in a runtime cache folder (e.g., under a temp dir) — do **not** commit or restructure.

> The named shapes are the *only* hard requirement the code relies on.

### 4.3 Slide builders (progressive, data‑gated)
Implement slide rendering as small functions in any module(s) you prefer. Each function should:
- Accept `(prs, data)` and append a slide.
- Fail gracefully if its subsection of `data` is missing — simply skip the slide.
- Never assume a layout index; find shapes by **name**.

Pseudo‑pattern (adapt as you like):
```python
def add_fleet_snapshot_slide(prs, data):
    if "summary" not in data: return False
    slide = add_slide_with_layout_named(prs, "fleet_snapshot")  # or however you prefer
    title = find_shape(slide, "Title"); title.text = "Fleet Snapshot"
    # Fill KPI_1..KPI_6 from data["summary"], formatting numbers consistently
    return True
```

### 4.4 Feature flag + dry‑run
- Guard the entire export behind a feature flag (env var or config), e.g., `FEATURE_EXPORT_PRELIM_DECK=true`.
- Support a **dry‑run** option that validates the JSON contract and template shapes, then reports which slides would be produced and why any were skipped — without writing a file.

### 4.5 Error handling
- If a required shape is missing, surface: `Missing shape 'KPI_1' on slide 'Fleet Snapshot'`. Offer remediation: "Open template and add a text box named KPI_1".
- If the template is not found and runtime template generation fails, return a typed error (don’t crash callers).

### 4.6 Logging (repo‑friendly)
- Use your existing logging. If none, keep prints minimal and prefix with `[prelim-deck]`.
- Log assumptions (e.g., MPG imputations) once per run, and surface them again on the **Data Gaps** slide.

## 5) Charting Rules (native PPTX)
- Use `python-pptx` native charts (`CategoryChartData`, `XL_CHART_TYPE`).
- Keep charts editable; no pasted raster images.
- Number formats: commas for thousands; currency `$#,##0`; tons `#,##0` + ` t`.
- Map to theme accents; **don’t hardcode RGB** unless your template lacks a theme.

## 6) Invocation patterns (you choose)
Pick the one that fits your app; none of these require new folders or renames.

- **CLI example:** wire a subcommand that parses a JSON path and optional template path, then calls `export_prelim_deck()`.
- **GUI button:** call `export_prelim_deck()` with the in‑memory dict; show a toast with the written path.
- **Pipeline hook:** after VIN decode + summaries, emit the JSON and call `export_prelim_deck()` if the feature flag is enabled.

**Output name recommendation** (not a rule): `{client_name}_{stage}_Prelim_{YYYY-MM-DD}.pptx` with spaces replaced by `_`.

## 7) Acceptance Criteria
- Deck builds with **no missing shape errors** given a valid JSON.
- Slides 1–5 always present when their sections exist; 6–13 appear only if data exists.
- Charts are editable in PPTX.
- Imputed data is footnoted on relevant slides and summarized in **Data Gaps**.
- End‑to‑end time < 10 s for ≤ 500 vehicles once JSON exists.
